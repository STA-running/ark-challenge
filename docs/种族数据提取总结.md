# 干员种族数据提取经验总结

> 撰写日期：2026-05-31  
> 项目：明日方舟自限挑战规则器  
> 背景：由于技术文档需要补充种族约束维度的可行性分析，为此从 PRTS 批量提取了所有干员的种族信息。

---

## 一、背景与动机

在项目 v3.1 的约束体系设计中，已经包含了 **阵营（nation/org）** 维度。但种族（race）作为明日方舟世界观中区分干员的另一个重要属性（如"萨卡兹""鲁珀""菲林"等），具备成为约束标签的潜力。因此需要确认：

1. PRTS 是否有结构化的种族数据
2. 能否批量获取所有干员的种族信息
3. 种族数据的质量和覆盖率如何

---

## 二、数据来源探查过程

### 2.1 探索路线

```mermaid
flowchart TD
    A[需要种族数据] --> B{PRTS Cargo API?}
    B -->|chara 表| C[无 race 字段]
    B -->|chara_extra_info 表| D[Cargo API 报 MWException]
    D --> E[表可能损坏或字段受限]
    B -->|chara_data 表| F[无 race 字段]
    A --> G{其他途径}
    G --> H[PRTS 页面 wikitext]
    H --> I[✅ 人员档案set 模板中有 |种族=xxx]
    G --> J[游戏数据 character_table.json]
    J --> K[❌ 无 race 字段，只有 nationId]
    G --> L[游戏数据 handbook_info_table.json]
    L --> M[✅ storyText 中有 【种族】xxx，但需要正则提取]
    G --> N[GitHub raw 直连]
    N --> O[❌ 国内网络超时，无法下载]
```

### 2.2 排查记录

| 数据源 | URL | 结果 | 耗时 |
|--------|-----|------|------|
| PRTS Cargo `chara` 表 | `/api.php?action=cargoquery&tables=chara&fields=chara.race` | ❌ 未知字段导致 MWException | 30s |
| PRTS Cargo `chara_extra_info` | `/api.php?action=cargoquery&tables=chara_extra_info` | ❌ 无论什么字段都报 MWException | 1min |
| PRTS Cargo `chara_data` | `/api.php?action=cargoquery&tables=chara_data` | ❌ 同上，报错 | 30s |
| PRTS 页面解析 API | `/api.php?action=parse&page=W&prop=wikitext` | ✅ 成功获取，模板中有 | 10s |
| PRTS 干员一览页面 | `/w/干员一览` | ❌ 动态加载，不含原始数据 | 20s |
| PRTS Cargo 表列表 | `/api.php?action=cargotables` | ✅ 看到所有 11 张表 | 10s |
| GitHub 游戏数据 | `raw.githubusercontent.com/.../handbook_info_table.json` | ❌ 国内直连超时 | 2min |
| GitHub 镜像 | `ghproxy.net/.../handbook_info_table.json` | ❌ 也超时 | 1min |

### 2.3 最终方案

**使用 MediaWiki 的 `action=query&prop=revisions&rvprop=content` API 批量拉取干员页面的 wikitext，从中用正则提取 `|种族=xxx` 字段。**

核心发现：
- PRTS 的 Cargo 表中**没有种族对应的结构化字段**
- 种族信息只存在于**每个干员页面本身的 wikitext** 的 `人员档案set` 模板中
- 同一信息也存在于游戏数据 `handbook_info_table.json` 的 `storyText` 中，但因网络限制不可用

## 三、技术实现

### 3.1 提取脚本

```python
# 脚本位置：scripts/extract_races_api.py
# 核心流程：

# Step 1: 从 Cargo chara 表获取所有干员名称
operators = api_get({
    "action": "cargoquery",
    "tables": "chara",
    "fields": "chara.cn,chara.charId",
    "where": "chara.charIndex > 0"
})

# Step 2: 批量获取 wikitext（每次 50 人）
wikitexts = api_get({
    "action": "query",
    "prop": "revisions",
    "rvprop": "content",
    "titles": "|".join(page_titles)  # 可同时查 50 页
})

# Step 3: 正则提取种族字段
race = re.search(r'\|\s*种族\s*=\s*([^\n|]+)', wikitext)
```

### 3.2 效率

| 指标 | 数值 |
|------|------|
| 总干员数 | 417 |
| 总 API 调用次数 | 9 次（8次×50人 + 1次×17人） |
| 总耗时 | ~20 秒 |
| 每批次间隔 | 1 秒（避免请求过快） |

### 3.3 输出格式

结果已保存为 `docs/operator_races.json`：

```json
{
  "char_113_cqbw": {
    "cn": "W",
    "race": "萨卡兹"
  },
  ...
}
```

也可通过 `race_stats` 字段直接查看按种族分组的人数汇总。

## 五、关键发现与教训

### 5.1 PRTS 数据结构的不一致性

PRTS 的数据存储方式**不是统一的**：

| 数据类型 | 存储位置 | 访问方式 |
|---------|---------|---------|
| 干员基础属性（职业、星级、标签） | Cargo `chara` 表 | 结构化查询 ✅ |
| 阵营信息（nation/team） | Cargo `chara` 表 | 结构化查询 ✅ |
| 种族信息 | 页面 wikitext 模板 | 需解析 ⚠️ |
| 详细档案文本 | 页面 wikitext 或游戏数据文件 | 需解析 ⚠️ |

**教训**：不要假设某个字段在 Cargo 表中就存在。Cargo 表只暴露了部分结构化字段，很多干员数据仍然只存在于页面模板中。

### 5.2 GitHub 国内访问问题

GitHub raw 文件的直连在国内**极其不稳定**（超时率 100%）。替代方案：
- 使用 MediaWiki API 直接爬取（已成功）
- 使用国内 CDN/镜像站（不一定可用）
- 考虑日本的 `ArknightsGameData_YoStar` 仓库

### 5.3 批量查询的效率优势

MediaWiki 的 `action=query` 支持**一次查询 50 个页面**的 wikitext，比逐个调用 `action=parse` 效率高 **50 倍**。417 人的数据只用 9 次 API 调用就完成了。

## 六、对项目的价值

### 6.1 可作为新的约束维度

```typescript
// 技术文档中可增加的约束类型
interface RaceConstraint {
  type: "race";
  race: string;           // 种族名，如"萨卡兹"
  mode: "allow" | "deny"; // 仅允许/仅禁止
}
```

### 6.2 可设计的标签建议

| 标签名 | 效果 | 可行性 |
|--------|------|--------|
| 萨卡兹之力 | 仅允许萨卡兹干员 | ✅ 29 人可选，足够组队 |
| 菲林成群 | 仅允许菲林干员 | ✅ 51 人，人数最多 |
| 黎博利飞行队 | 仅允许黎博利 | ✅ 42 人 |
| 稀有种族挑战 | 仅允许种族人数 ≤5 的干员 | ✅ 19 种稀有，有深度 |
| 大熔炉 | 禁止人数最多的 5 个种族 | ✅ 有平衡性 |
| 异族集结 | 队内不能有相同种族 | ✅ 有挑战性 |
| 未公开档案 | 仅允许种族为"未公开"的干员 | ✅ 24 人极限挑战 |

### 6.3 数据文件用途

`docs/operator_races.json` 可作为项目中 `race` 字段的数据源，在构建干员数据 JSON 时合并进去。

## 七、未来维护要点

1. **更新频率**：种族信息几乎不会随版本更新而改变（联动角色可能有新种族），数据**长期有效**
2. **验证方式**：如果鹰角更新了游戏数据，需要重新跑一次脚本，把 `operator_races.json` 合并到项目数据中
3. **脚本复用**：`scripts/extract_races_api.py` 可以直接复用，只需修改输出路径
4. **联动角色**：联动角色（彩虹小队、Ave Mujica、迷宫饭）的种族多为"未公开"或特殊名称，需要注意
