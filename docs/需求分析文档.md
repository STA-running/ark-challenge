## 一、核心认知（最重要的一段）

### 1.1 我们做什么、不做什么

| | 能做的 ✅ | 不能做的 ❌ |
|---|---------|-----------|
| 本质 | **自律挑战规则生成器** — 帮玩家制定规则，自觉遵守 | ~~游戏修改器~~ — 我们改不了游戏内任何数据 |
| 编队构建 | 限制自己选什么干员、带几个人、什么职业 | ~~修改干员属性~~ — 费用、攻击力改不了 |
| 行为自律 | 声明"我不开技能""我不撤退"等规则 | ~~强制执行~~ — 只能靠自觉，游戏里没东西拦你 |
| 关卡机制 | 选择挑战哪个关卡 | ~~修改关卡机制~~ — 费用回复速度、敌方血量、地块类型改不了 |
| 价值 | 帮你**想清楚规则** + **验证编队合规** + **分享挑战** | 不能帮你**自动执行**规则 |

> 💡 **一句话定位**：这不是外挂，是"挑战规则的笔记本 + 编队合规检查器"。
> 就像跑步给自己定目标"今天跑5公里"——手表只能记录你跑了多少，跑不跑靠你自己。

### 1.2 目标用户

| 用户类型 | 画像 | 核心诉求 |
|---------|------|---------|
| 自限玩家 | 觉得正常玩法太简单，想给自己加难度 | 生成并管理挑战规则，验证编队合规 |
| 少人/低配党 | B站/NGA社区常见的极限挑战玩家 | 快速筛选合规干员，导出分享码 |
| 攻略作者 | 写低配/少人攻略的内容创作者 | 将挑战规则编码为可分享的链接/码 |
| 休闲玩家 | 想尝试不同玩法但不知从何下手 | 使用内置预设模板，快速开始 |

## 三、约束体系设计（核心 — Q4最终答案）

### 3.1 设计原则

约束体系分**两大类**，处理逻辑完全不同：

| 类型 | 定义 | 工具能做什么 | 举例 |
|------|------|-------------|------|
| **硬约束** | 可以在编队阶段机械验证的规则 | ✅ 自动过滤/验证/标红 | 禁用职业、限制人数、稀有度上限 |
| **软约束** | 只能声明，靠玩家在游戏中自觉遵守 | ⚠️ 只能展示为规则卡片，无法验证 | 不开技能、挂机、不撤退 |

> 核心思路：**硬约束交给工具验证，软约束交给玩家自律。工具不装能验证不了的东西。**

### 3.2 硬约束清单（工具可验证）

这些约束都能通过检查编队数据来判断是否合规：

#### A. 编队规模约束

| 约束ID | 约束名 | 类型 | 默认值 | 说明 |
|--------|--------|------|--------|------|
| `maxSquadSize` | 编队人数上限 | 数值 | 12 | 限制编队中干员数量（1~12） |
| `maxSameProfession` | 同职业上限 | 数值 | 无限制 | 同一职业最多带几个 |
| `minProfessionCount` | 某职业最少人数 | 键值对 | 无限制 | 如"至少带1个医疗" |

#### B. 职业与分支约束

| 约束ID | 约束名 | 类型 | 默认值 | 说明 |
|--------|--------|------|--------|------|
| `allowedProfessions` | 允许职业 | 多选 | 全部 | 白名单模式：只允许这些职业 |
| `bannedProfessions` | 禁用职业 | 多选 | 无 | 黑名单模式：禁用这些职业 |
| `allowedSubProfessions` | 允许分支 | 多选 | 全部 | 如只允许"处决者" |
| `bannedSubProfessions` | 禁用分支 | 多选 | 无 | 如禁用"投降先锋" |

> 💡 白名单和黑名单互斥：选了白名单则黑名单失效，反之亦然。

#### C. 稀有度约束

| 约束ID | 约束名 | 类型 | 默认值 | 说明 |
|--------|--------|------|--------|------|
| `maxRarity` | 最高稀有度 | 数值 | 6 | 如 maxRarity=3 表示只能用1~3星 |
| `minRarity` | 最低稀有度 | 数值 | 1 | 如 minRarity=5 表示只能用5~6星 |
| `maxSixStarCount` | 6星数量上限 | 数值 | 无限制 | 如最多带1个6星 |

#### D. 费用约束

| 约束ID | 约束名 | 类型 | 默认值 | 说明 |
|--------|--------|------|--------|------|
| `maxTotalCost` | 编队总费用上限 | 数值 | 无限制 | 编队所有干员部署费用之和≤此值（以精二费用为准） |
| `maxSingleCost` | 单干员费用上限 | 数值 | 无限制 | 禁止编入费用超过此值的干员（以精二费用为准） |
| `minSingleCost` | 单干员费用下限 | 数值 | 无限制 | 禁止编入费用低于此值的干员（以精二费用为准） |

#### E. 干员黑白名单

| 约束ID | 约束名 | 类型 | 默认值 | 说明 |
|--------|--------|------|--------|------|
| `allowedOperators` | 允许干员 | ID列表 | 全部 | 白名单：只允许这些干员 |
| `bannedOperators` | 禁用干员 | ID列表 | 无 | 黑名单：禁用这些干员 |

#### F. 干员属性约束

| 约束ID | 约束名 | 类型 | 默认值 | 说明 |
|--------|--------|------|--------|------|
| `trustOnly` | 仅信赖干员 | 布尔 | false | 只能用非限定/常驻干员 |
| `positionRestriction` | 放置位置限制 | 多选 | 无限制 | 如"只能近战位"/"只能远程位" |

#### G. 阵营约束（🆕 v3.1 新增）

> 💡 来源：PRTS Cargo API `chara.nation`（国家/地区）和 `chara.team`（组织/团队）字段。
> nation 覆盖率约90%（粗粒度），org 覆盖率约10%（细粒度），两者互补。

| 约束ID | 约束名 | 类型 | 默认值 | 说明 |
|--------|--------|------|--------|------|
| `sameNationOnly` | 仅同国家/地区 | 布尔 | false | 编队中有 nation 的干员必须属于同一国家/地区，无 nation 的干员不参与约束 |
| `sameOrgOnly` | 仅同组织/团队 | 布尔 | false | 编队中有 org 的干员必须属于同一组织，无 org 的干员不参与约束 |
| `allowedNations` | 允许国家/地区 | 多选 | 全部 | 白名单：只允许这些国家/地区（如"炎-龙门""乌萨斯"） |
| `bannedNations` | 禁用国家/地区 | 多选 | 无 | 黑名单：禁用这些国家/地区 |
| `allowedOrgs` | 允许组织/团队 | 多选 | 全部 | 白名单：只允许这些组织（如"莱茵生命""彩虹小队"） |
| `bannedOrgs` | 禁用组织/团队 | 多选 | 无 | 黑名单：禁用这些组织 |

> ⚠️ **设计要点**：nation 和 org 覆盖率差异大。"仅同国家/地区"挑战最实用（90%干员有归属），
> "仅同组织/团队"挑战更有趣但限制更严（仅10%干员有组织归属，其他人不参与约束）。
> 白名单/黑名单互斥规则同职业约束。

#### H. 种族约束（🆕 v3.2 新增）

> 💡 **数据来源**：PRTS 干员页面的 wikitext "人员档案set" 模板中的 `|种族=xxx` 字段，非 Cargo API 结构化数据。
> 已通过 PRTS API 批量提取完成，结果保存为 `docs/operator_races.json`。
> 覆盖 409/417 名干员（98.1%），44 个种族种类。无种族的 8 人全为机器人单位。

| 约束ID | 约束名 | 类型 | 默认值 | 说明 |
|--------|--------|------|--------|------|
| `allowedRaces` | 允许种族 | 多选 | 全部 | 白名单：只允许这些种族的干员（如 "萨卡兹" "菲林"） |
| `bannedRaces` | 禁用种族 | 多选 | 无 | 黑名单：禁用这些种族的干员 |
| `sameRaceOnly` | 仅同种族 | 布尔 | false | 编队中所有有种族信息的干员必须属于同一种族，无种族的干员不参与约束 |
| `rareRaceOnly` | 仅稀有种族 | 布尔 | false | 仅允许人数少于此阈值的种族的干员（如只使用全服仅1人的种族） |

> ⚠️ **设计要点**：
> - 种族共 44 种，但大型种族（菲林51人、黎博利42人、萨卡兹29人）和微型种族（1-2人）差距悬殊，**标签设计需要平衡**，不能让大种族太容易、小种族太难组队
> - `rareRaceOnly` 的阈值建议设为 **≤5人**，这样覆盖 19 个种族共约 40 人，组队难度较高但有挑战空间
> - 白名单/黑名单互斥规则同职业约束
> - `sameRaceOnly` 与 `sameNationOnly`/`sameOrgOnly` 类似，但更通用（覆盖率 98% vs 90%/10%）

#### I. 阻挡数约束（🆕 v3.3 新增）

> 💡 **阻挡数规则**：以干员**面板数据**为准，不考虑技能效果。因为游戏中许多干员开技能后会改变阻挡数（如塞雷娅钙质化+1阻挡），但面板阻挡数才是自限挑战的判定基准。
> 数据来源：PRTS 干员页面 wikitext "人员档案set" 模板中的 `|阻挡数=1` 字段，100%覆盖。
> 共分四档：0阻挡（7人）、1阻挡（281人）、2阻挡（103人）、3+阻挡（26人）。

| 约束ID | 约束名 | 类型 | 默认值 | 说明 |
|--------|--------|------|--------|------|
| `blockTier` | 阻挡数限制 | 枚举 | 无限制 | 限制编队中干员的阻挡数档位，可选 0 / 1 / 2 / 3+ |
| `blockTierMode` | 阻挡数模式 | 枚举 | `exact` | `exact`=所有干员阻挡数必须完全等于指定档位；`max`=不超过指定档位 |
| `maxSameBlockTier` | 同阻挡数上限 | 数值 | 无限制 | 编队中同一阻挡数档位的干员人数上限 |

> ⚠️ **设计要点**：
> - `exact` 模式最严格：选"仅1阻挡"则编队里只能有阻挡数为1的干员（281人可选）
> - `max` 模式稍宽松：选"不超过1阻挡"则0挡和1挡都可以用（288人）
> - `maxSameBlockTier` 可用于"阻挡多样性"挑战，如"任何阻挡数档位不超过4人"强制职业混合
> - 阻挡数约束与位置约束有天然关联：远程位基本全1挡，近战位分布广

这些约束**无法通过编队数据验证**，但可以作为"挑战规则"声明，展示在方案卡片上：

| 约束ID | 约束名 | 类型 | 说明 |
|--------|--------|------|------|
| `noSkill` | 禁用主动技能 | 布尔 | 不手动开启任何干员技能 |
| `noAutoSkill` | 禁止自动技能干员 | 布尔 | 🆕 禁止编入有自动触发技能的干员（自动释放干员会自行触发，无法靠自律控制） |
| `afkOnly` | 仅挂机 | 布尔 | 部署后不进行任何操作 |
| `noRetreat` | 禁止撤退 | 布尔 | 干员上场后不主动撤退 |
| `noRedeploy` | 禁止再部署 | 布尔 | 每个干员只能部署一次 |
| `zeroLeak` | 零漏击 | 布尔 | 任何敌人不能进入蓝色目标点 |
| `maxSteps` | 操作步数上限 | 数值 | 华莱士流：最多操作N步 |
| `customRule` | 自定义规则 | 文本 | 自由填写，如"只使用2023年前实装的干员" |

> 💡 软约束在UI上用虚线框/斜体区分，标注"🤝 自律规则 — 工具无法验证，请自觉遵守"

### 3.4 关卡关联（可选）

虽然我们不能修改关卡机制，但可以把方案**关联到具体关卡**：

| 字段 | 说明 |
|------|------|
| `stageId` | 关卡ID（如"main_01-07"） |
| `stageName` | 关卡名称（如"1-7"） |
| `difficulty` | 难度（普通/突袭/磨难） |

这样在分享方案时，别人就知道"这个挑战是针对哪个关卡的"。

## 四、功能需求

### 4.1 P0 核心功能

#### F0 · 挑战标签系统（核心中的核心 — 本次新增）

> 💡 **设计理念**：把每种自限玩法封装成一个**标签**，用户可以**随机抽取**或**手动选择**标签来组合挑战方案。就像游戏里的标签招募——选标签，出方案！

**标签是什么？**

每个标签 = 一个自限玩法的封装：
- 一个自解释的名称（如"🏃 编队≤4"），名字就是规则+指标值，一目了然
- 一段详细说明（如"本挑战限制编队人数上限为4人（不含助战单位），进入关卡后可部署干员数不超过4"），消除歧义
- 自动映射到具体的硬约束/软约束配置

**两种使用方式：**

| 方式 | 说明 | 场景 |
|------|------|------|
| 🎲 **随机抽奖** | 点击按钮，随机抽取2~4个标签组合成挑战方案 | 想玩但不知道玩什么，让命运决定！ |
| 🏷️ **手动选标签** | 从标签池中勾选想要的标签，自由组合 | 心里有想法，精确控制规则 |

**标签抽奖交互设计：**

```

│                                               │
│   抽取数量:  [2] [3] [4]                      │
│                                               │

│   └──┬───┘  └──┬───┘  └──┬───┘               │
│      │         │         │                    │
│      ▼         ▼         ▼                    │
│   🔄换一个 🔄换一个 🔄换一个                   │
│   🚫拉黑   🚫拉黑   🚫拉黑                    │
│   ↑ 点击后该标签不再出现在抽奖和标签池中          │

│   [🎲 全部重抽]  [✅ 采用此方案]               │
```

#### F2 · 合规编队构建器

- **页面结构**：仅两个页面（编队页 + 设置页），导航栏精简为两个Tab
- **编队页布局**：左右分栏
  - **左栏（flex:1）**：编队网格 + 干员选择区
  - **右栏（260px固定）**：挑战标签 + 关卡 + 方案管理

- **操作流程**：
  1. 先在右侧选择挑战标签（手动勾选或随机抽取）→ 点击「✅ 确认标签」
  2. 再选择关卡（可换一个）→ 点击「✅ 确认关卡」
  3. 确认后，干员选择区自动启用，开始编队

- **已确认方案横幅**：在编队网格上方显示已确认的标签和关卡信息（如"🏷️ 编队≤4 · ★≤3 | 🗺️ 4-10"）

- **标签选择**：
  - 两种模式切换：`[📝 手动选]` 和 `[🎲 随机抽]`
  - 手动模式：展示所有标签带checkbox，勾选即选中
  - 随机模式：随机抽取2~4个标签，可点击「🔄 重新抽」刷新
  - 选择后需点击「✅ 确认标签」才生效

- **编队网格布局**：2行×6列 CSS Grid，最多容纳12人
  - 填充顺序：**列优先**（先填(行1,列1)→(行2,列1)→(行1,列2)→(行2,列2)→...）
  - 左对齐、靠上对齐：干员数少于12时，右边和下边直接留白，不渲染空位占位元素
  - 每个干员以**头像卡片**展示：圆形头像框(40px) + 干员名 + 职业名，固定86×96px
  - 头像优先显示从PRTS拉取的60px缩略图，加载失败时回退为职业图标
  - 悬停显示右上角 ✕ 删除按钮

- **干员选择区**（左栏下方，初始禁用，确认标签后启用）：
  - 按 **8 大职业**（先锋/近卫/重装/狙击/术师/医疗/辅助/特种）分类浏览
  - 按 **⭐稀有度**（1~6星）筛选：点击稀有度先选星级，再展示该星级全部干员
  - 按 **👿种族**筛选：点击种族先选种族名，再展示该种族全部干员
  - 搜索框支持按名称搜索干员
  - **单击干员按钮 → 展示干员信息预览**（职业、稀有度、种族、费用、阻挡数等标签相关面板数值）

  - 已选的干员自动从选择区消失（防重复选择）
  - 从编队网格移出干员后，干员重新出现在选择区

  - 编队人数超限 → 标红 + 提示
  - 选中了被禁职业的干员 → 标红 + 提示
  - 总费用超限 → 标红 + 提示
  - 稀有度不符 → 干员灰显不可选
  - 阵营不符（选了非同国家/非同组织的干员）→ 标红 + 提示（🆕 v3.1）
  - 种族不符（选了被禁种族的干员、非同种族的干员、超出稀有种族限制的干员）→ 标红 + 提示（🆕 v3.2）

#### F4 · 分享码（🆕 超短编码）

- 将挑战方案导出为**超短分享码**（目标：15~30字符）
- 通过分享码导入他人方案
- 分享码可复制到剪贴板

**设计思路：分享规则，不分享阵容**

分享码的核心逻辑是：**只编码标签+关卡+软约束开关**，不编码具体干员选了谁。因为挑战的本质是规则，你选什么干员是自己根据规则决定的。

```
编码流程：
  选中标签ID列表 → 20位位掩码(3字节) 
  + 关卡ID(2字节)
  + 软约束开关(1字节)  
  + 手动覆盖标记(1字节)
  → 二进制拼接 → Base64URL编码 → AKRL:前缀

典型分享码：AKRL:GwYhABoB（仅8字符载荷，含前缀14字符）
```

| 编码内容 | 位数 | 说明 |
|---------|------|------|
| 标签选择 | 20 bits | 20个标签，1个bit=1个标签选中/未选中 |
| 关卡ID | ~16 bits | 关卡表索引，支持65535个关卡 |
| 软约束开关 | 6 bits | 6个软约束各1bit |
| 手动覆盖标记 | 1 bit | 是否有手动修改过约束 |
| 版本号 | 4 bits | 预留版本兼容 |
| **合计** | **~47 bits ≈ 6 bytes** | Base64URL后约 **8字符** |

> 💡 如果用户手动修改过约束值（比如标签默认编队≤4，手动改成≤6），则附加一个精简的覆盖数据段，分享码会稍长但仍控制在30字符以内。

#### F7 · 方案验证报告

- 一键检查编队是否满足所有硬约束

## 八、风险与假设

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 软约束无法验证 | 用户可能在游戏中不遵守 | UI明确标注"🤝自律规则"，不假装能验证 |
| 干员数据随版本更新 | 需要持续维护 | 爬虫脚本一键更新 |
| 游戏素材版权 | 不能直接分发游戏资源 | 仅使用文字+职业图标 |
| 用户对"硬/软约束"概念理解成本 | 可能混淆 | UI用实线/虚线、颜色、标注区分 |
| 精英化不可逆导致费用计算偏差 | 部分玩家干员未精二，实际费用更低 | UI标注"费用以精二为准"，elite0/elite1作参考展示 |
| 关卡分类体系 | 需设计合理分类 | 类型×难度双维度：普通/Boss/超难/超难Boss × 标准/突袭/沙盘推演 |

## 九、术语表

| 术语 | 解释 |
|------|------|
| 硬约束 | 可通过编队数据机械验证的规则，如"禁用医疗" |
| 软约束 | 只能声明、靠玩家在游戏中自觉遵守的规则，如"不开技能" |
| 干员 | 玩家可部署的角色单位 |
| 编队 | 玩家带入关卡的干员组合，最多12人 |
| 部署费用 / DP | 部署干员需消耗的点数，战斗中自然回复 |
| 精英化 | 干员强化阶段（精0/精1/精2），**不可逆**，精二后无法回退，影响费用和属性 |
| 部署费用计算规则 | 本工具统一以**精二状态**的部署费用为准；如玩家某干员尚未精二，实际费用会更低 |
| 子分支 | 职业下的细分类型，如"近卫-剑豪" |
| 少人挑战 | 限制编队人数的自限玩法 |
| 低配挑战 | 限制干员稀有度的自限玩法 |
| 华莱士流 | 以最少操作步数+最早完成时间为目标的挑战流派 |
| 分享码 | 编码后的挑战方案字符串，便于文本分享 |
| 标签/Tag | 自限玩法的封装单元，标签名含规则+指标值（如"编队≤4""总费≤80""★≤3"），detail字段提供消除歧义的详细说明 |
| 标签详细说明/detail | 标签的完整语义解释，补充名字无法涵盖的边界情况，如"不含助战单位""以精二费用为准"等 |
